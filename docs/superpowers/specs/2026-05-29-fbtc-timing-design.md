# FBTC Market Timing Dashboard — Design Spec

**Date:** 2026-05-29
**Author:** Guillaume Monjal
**Status:** Approved

---

## Goal

A daily-updated static web dashboard (GitHub Pages) that tells whether now is a good time to make a lump-sum entry into FBTC ETF within an RRSP/TFSA. The tool is cycle-aware and long-term oriented: it aims to identify accumulation zones near Bitcoin cycle bottoms and avoid entries near cycle tops.

---

## Architecture

```
GitHub Actions (daily cron, 8am ET)
  │
  ├── fetch_data.py        → raw price + on-chain data (CSV/JSON, cached in repo)
  ├── compute_signals.py   → 6 signal readings from raw data
  ├── score.py             → weighted composite score (0–100)
  └── build_dashboard.py   → renders Jinja2 template → docs/index.html → pushed to gh-pages

backtest.py                → run manually to re-derive weights.json
```

Three layers:
- **Data** — fetched from free public APIs, cached as CSV/JSON in the repo
- **Signal + Score** — pure Python computation, no external dependencies at runtime
- **Dashboard** — single self-contained HTML file generated by a Jinja2 template, hosted on GitHub Pages

## Signals

Six signals, all proven cycle indicators:

| Signal | Buy Zone | Neutral Zone | Avoid Zone |
|---|---|---|---|
| **MVRV Z-Score** | < 1 | 1–3 | > 3 |
| **200-Week Moving Average** | Price below 200WMA | Price 0–20% above | Price > 20% above |
| **Monthly RSI** | < 40 | 40–60 | > 70 |
| **Pi Cycle Top/Bottom** | 111DMA below 2×350DMA | Converging | 111DMA crosses above 2×350DMA |
| **Puell Multiple** | < 0.5 | 0.5–1.5 | > 1.5 |
| **Fear & Greed Index** | < 25 (Extreme Fear) | 25–50 | > 50 |

Each signal outputs a score of 0 (avoid), 50 (neutral), or 100 (buy zone). The composite score is the weighted average of all six.

---

## Backtesting & Weighting

**Definition of a good entry:** A day in Bitcoin history where:
- Buying BTC and holding for 18 months produces ≥ 50% return, AND
- The entry price is within the bottom 40% of that cycle's range

**Methodology:** For each signal, scan every day in Bitcoin history (2013–present):
1. Check if the signal was in its buy zone that day
2. Check if that day met the good-entry definition
3. Compute **precision** (signal said buy → was it a good entry?) and **recall** (all good entries → did signal catch it?)
4. Derive weight from F1 score — higher precision → higher weight

Output: `data/weights.json`, committed to the repo. The daily job reads this file — no re-backtesting on every run.

**Re-running the backtest:** `backtest.py` can be triggered manually via GitHub Actions workflow dispatch, or run locally, whenever the methodology needs updating.

---

## Composite Score & Verdicts

```
Composite = Σ (signal_score × signal_weight)
```

| Score | Verdict |
|---|---|
| 75–100 | **STRONG BUY** — multiple signals align, historically excellent entry |
| 50–74 | **ACCUMULATE** — conditions favorable but not peak conviction |
| 25–49 | **WAIT** — mixed signals, patience recommended |
| 0–24 | **AVOID** — historically poor entry conditions |

## Dashboard

Single self-contained HTML page, generated daily, hosted on GitHub Pages.

**Visible at all times:**
- BTC current price + last updated timestamp
- Composite score (large, color-coded)
- Verdict label (STRONG BUY / ACCUMULATE / WAIT / AVOID)
- Signal table: name, current reading, status (green/yellow/red)
- Historical chart: composite score vs BTC price (2013–present), built with Chart.js via CDN

**Hidden by default (expandable):**
- Methodology section: signal weights, backtest precision/recall per signal, good-entry definition

**Tech:** Plain HTML + CSS + Chart.js (CDN). No JavaScript framework. Jinja2 template rendered by Python build script. Zero hosting dependencies beyond GitHub Pages.

## GitHub Actions — Daily Workflow

Schedule: `0 12 * * *` (8am ET / 12:00 UTC)

Steps:
1. Checkout repo
2. Set up Python + install `requirements.txt`
3. `python scripts/fetch_data.py` — refresh CSVs
4. `python scripts/compute_signals.py` — compute signal readings
5. `python scripts/score.py` — apply weights, output `data/current_score.json`
6. `python scripts/build_dashboard.py` — render `docs/index.html`
7. Commit + push `data/` and `docs/index.html`

---

## Out of Scope

- No sell signals — this tool is for entry timing only
- No FBTC-specific price tracking (uses BTC as proxy, which is what FBTC tracks)
- No notifications or alerts — the dashboard is the interface
- No account integration — purely informational, no brokerage API
